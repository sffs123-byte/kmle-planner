# 국시 플래너 — 자비스 전공의 + 자비스 교수님 협업 운영안

## 결론
**세 명(강렬 + 자비스 전공의 + 자비스 교수님) 동시 그룹채팅은 2단계 옵션**으로 두고,
실제 운영은 우선 **숨은 협업(hidden collaboration)** 구조로 가는 것이 가장 안정적이다.

즉,
- 강렬은 가능하면 **하나의 창구**만 본다.
- 내부적으로는
  - `자비스 전공의(clerk)`가 실습 일정/생활 제약을 관리하고
  - `자비스 교수님(academic)`이 국시 우선순위/복습 로직을 관리하고
  - 메인 자비스 또는 플래너 owner가 둘을 합쳐 최종 일정에 반영한다.

---

## 왜 그룹채팅을 1순위로 두지 않는가
그룹채팅은 보기엔 직관적이지만 실사용에서는 아래 문제가 생긴다.

1. **중복 응답**
   - 둘 다 같은 메시지에 반응할 수 있음
2. **책임 경계가 흐려짐**
   - 누가 최종 결정을 내리는지 애매해짐
3. **플래너 데이터 일관성 깨짐**
   - 전공의가 일정 수정, 교수님이 우선순위 수정, 메인 앱 반영 순서가 꼬일 수 있음
4. **대화는 풍성하지만 실제 운영은 지저분**해지기 쉬움

따라서 그룹채팅은
- 데모용
- 디버깅용
- 특정 의사결정 회의용
정도로만 쓰는 게 좋다.

---

## 추천 구조: 숨은 협업 + 단일 사용자 창구

### 1) 자비스 전공의 (clerk)
**소유 영역:** 현실 제약과 실습 일정

관리할 정보:
- 실습 일정
- 병원 출근/이동/회의/당직 등 고정 블록
- 피곤한 시간대, 사용 가능 시간대
- 갑자기 비는 시간 / 갑자기 막히는 시간
- 일정 충돌 여부

전공의가 주는 출력 예:
- 이번 주 공부 가능한 시간 블록
- 절대 못 건드리는 일정
- 하루 최대 학습 가능량
- 실습 후 가벼운 복습만 가능한 날 표시

### 2) 자비스 교수님 (academic)
**소유 영역:** 국시 전략과 학습 우선순위

관리할 정보:
- 과목 우선순위
- 단계별 목표량 (강의/개념틀/문풀/오답/복습)
- 반복 복습 원칙
- 약한 과목 교정 우선순위
- 이번 주 핵심 과목/파트

교수님이 주는 출력 예:
- 이번 주 최우선 과목 3개
- 각 과목에서 지금 밀리고 있는 단계
- 복습 주기 추천
- 실습 일정 속에서도 유지할 최소 학습 루프

### 3) 메인 자비스 또는 플래너 owner
**소유 영역:** 최종 반영 및 앱 상태 관리

할 일:
- 전공의 입력 = hard constraint로 반영
- 교수님 입력 = soft priority로 반영
- 둘을 합쳐 최종 일정 생성
- 충돌 시 강렬에게 "선택지" 형태로 보여주기
- 실제 플래너 상태 저장/수정

---

## 우선순위 규칙
플래너 반영 시 기본 룰은 이렇게 두는 게 좋다.

### Hard constraints (전공의 우선)
반드시 지켜야 하는 것:
- 실습 시간
- 이동 시간
- 회의/과제/병원 일정
- 수면/회복이 필요한 구간

### Soft priorities (교수님 우선)
가능한 시간 안에서 최적화할 것:
- 어떤 과목을 먼저 할지
- 어떤 단계가 먼저인지
- 복습을 어디 넣을지
- 어떤 과목을 이번 주에 최소한 어디까지 밀어야 하는지

즉,
**전공의가 시간의 바깥 틀을 만들고, 교수님이 그 안의 공부 전략을 채운다.**

---

## 실무 운영 프로토콜

### 매주 초
1. 전공의가 이번 주 일정 제약 정리
2. 교수님이 이번 주 국시 우선순위 정리
3. 메인 자비스가 플래너에 배치
4. 강렬에게 이번 주 계획안 제시

### 매일 저녁 또는 아침
1. 전공의가 내일 일정 변동 확인
2. 교수님이 복습/취약 과목 보정 필요 여부 확인
3. 메인 자비스가 내일 일정 재배치

### 공부 후
1. 강렬이 실제 수행량 입력
2. 교수님이 단계 진행도 재평가
3. 전공의가 일정 부담도를 다시 판단

---

## 그룹채팅을 만들고 싶다면 (2단계 옵션)
가능은 하다. 다만 반드시 **멘션 기반 규칙**이 필요하다.

예시:
- `@자비스 전공의` → 실습 일정/현실 시간표 질문만
- `@자비스 교수님` → 국시 우선순위/학습 전략 질문만
- `@총괄 자비스` → 최종 플래너 반영 요청만

이렇게 해야 혼선이 줄어든다.

### 그룹채팅을 쓸 때의 좋은 용도
- 특정 주간 계획 회의
- 실습과 국시가 충돌할 때 조정 회의
- "이번 주는 현실적으로 어디까지 가능한가" 합의할 때

### 그룹채팅을 상시 운영으로 두지 않는 이유
- 대화는 풍부해도 실제 state update가 꼬이기 쉽다.
- 최종 상태를 누가 저장했는지 불명확해진다.

---

## 실제 핸드오프 패킷
자비스 전공의에게 줄 것:
- 이번 주 실습 일정표
- 고정 일정 / 유동 일정 / 피곤도 정보
- 공부 가능한 시간창 정리 요청

자비스 교수님에게 줄 것:
- 현재 과목별 진행도
- 취약 과목/파트
- 이번 주 남은 시간
- 우선과목/복습 루프 추천 요청

메인 자비스가 최종적으로 들고 있어야 할 것:
- `kmle-planner/index.html`
- 현재 플래너 데이터
- 최종 반영 규칙
- 충돌 해결 정책

---

## 강렬에게 가장 편한 사용자 경험
강렬은 가능하면 아래처럼 쓰는 게 제일 편하다.

### 평소
- 메인 자비스에게만 말함
- 메인 자비스가 내부적으로 전공의/교수님 의견을 받아와 반영

### 특별한 상황
- 이번 주 실습이 빡세서 일정 조정이 필요할 때 → 전공의와 직접
- 국시 전략을 깊게 조정하고 싶을 때 → 교수님과 직접

즉,
**평소에는 단일 창구, 중요한 조정 때만 전문가 직접 호출** 구조가 좋다.

---

## 바로 실행할 추천안
### 추천안 A (가장 현실적)
- 강렬 ↔ 메인 자비스
- 메인 자비스 ↔ 자비스 전공의 / 자비스 교수님
- 필요할 때만 각 전문가 호출

### 추천안 B (보여주기용)
- 텔레그램 그룹 하나 생성
- 강렬 + 전공의 + 교수님 + 총괄 자비스
- 멘션 기반으로만 응답
- 상시 운영이 아니라 조정 회의용

---

## 한 줄 정리
**채팅방을 반드시 만들 필요는 없다.**
오히려 처음에는
- 전공의 = 현실 일정 owner
- 교수님 = 국시 전략 owner
- 메인 자비스 = 최종 통합 owner
구조로 가는 것이 가장 안정적이다.
